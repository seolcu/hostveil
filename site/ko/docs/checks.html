    점검 항목

    hostveil은 셀프호스터가 실제로 침해당하는 영향이 가장 큰 경로들을 점검한 뒤, 모든 것을 하나의 0–100 점수로 합칩니다. 각 발견 항목은 domain.rule 형태의 고정된 ID를 가지며, explain, fix, rollback과 함께 사용합니다.

    Docker나 Trivy가 없나요? 해당 영역은 깔끔하게 건너뛰어지고 점수가 재정규화되므로, 실행되지도 않은 스캔에 대해 오해를 살 만큼 완벽한 결과를 받는 일은 없습니다.

    심각도와 점수

    모든 발견 항목은 네 가지 심각도 중 하나를 가집니다. 점수는 100에서 시작하며, 수정되지 않은 발견 항목마다 심각도별 감점이 적용됩니다.

    심각도감점
    심각8
    높음5
    보통2
    낮음1

    hostveil scan은 수정되지 않은 발견 항목 중 심각 또는 높음이 하나라도 있으면 0이 아닌 값으로 종료합니다 — CI나 cron 게이트로 유용합니다.

    SSH ssh.

    sshd_config에서 직접 파싱합니다. 읽으려면 root 권한이 필요하며, hostveil은 sudo로 자동 상승하여 이를 얻습니다.

    ID무엇을 표시하는지심각도수정
    ssh.emptypasswords빈 비밀번호가 허용됨심각자동 수정
    ssh.rootlogin비밀번호를 사용한 root 로그인이 허용됨높음검토
    ssh.passwordauth비밀번호 인증이 허용됨보통검토
    ssh.maxauthtries연결당 인증 시도 횟수가 너무 많음낮음자동 수정
    ssh.x11forwardingX11 포워딩이 활성화됨낮음자동 수정

    Docker / Compose compose.

    Compose 파일에 대한 네이티브 감사입니다 — 외부 스캐너가 필요 없습니다.

    ID무엇을 표시하는지심각도수정
    compose.ds016Docker 소켓이 컨테이너에 마운트됨심각수동
    compose.ds018데이터 저장소가 모든 인터페이스에 노출됨심각자동 수정
    compose.ds001컨테이너가 특권 모드로 실행됨높음수동
    compose.ds019관리자 패널이 모든 인터페이스에 노출됨높음자동 수정
    compose.dr001컨테이너가 호스트 네트워킹 모드를 사용함높음수동
    compose.ds017민감한 호스트 경로가 읽기-쓰기로 마운트됨높음수동
    compose.ds005위험한 리눅스 capability를 추가함높음수동
    compose.dr005환경 변수에 하드코딩된 비밀값높음수동
    compose.ds006no-new-privileges 강화 설정 누락보통자동 수정
    compose.ds009컨테이너가 root로 실행됨보통수동
    compose.dr002포트가 모든 인터페이스에 게시됨보통자동 수정
    compose.ds008재시작 정책이 설정되지 않음낮음자동 수정
    compose.ds010메모리 제한이 설정되지 않음낮음검토
    compose.ds012헬스체크가 정의되지 않음낮음수동
    compose.dr004env_file에서 비밀값을 불러옴낮음수동

    방화벽 firewall.

    ID무엇을 표시하는지심각도수정
    firewall.inactive활성화된 호스트 방화벽이 없음 (ufw, firewalld, 또는 nftables)높음수동

    자동 업데이트 updates.

    ID무엇을 표시하는지심각도수정
    updates.disabled자동 보안 업데이트가 활성화되지 않음 (apt unattended-upgrades 또는 dnf-automatic)보통검토

    노출된 서비스 ports.

    ss로 호스트의 수신 대기 중인 TCP 소켓을 읽어, 루프백이 아닌 주소에 바인딩된 서비스를 표시합니다 — 컨테이너가 아니기 때문에 Compose 파일 점검으로는 볼 수 없는, 호스트에 직접 설치된 데이터베이스·관리 패널·앱입니다. 루프백 전용 바인딩은 무시하며, SSH는 정상으로 간주해 여기서 표시하지 않습니다.

    ID표시 대상심각도수정
    ports.exposed-datastore데이터스토어(Postgres, MySQL, Redis, MongoDB 등)가 네트워크에서 접근 가능높음수동
    ports.exposed-admin관리 UI(예: Portainer)가 네트워크에서 접근 가능높음수동
    ports.exposed다른 서비스가 루프백이 아닌 주소에서 수신 대기 중이고 활성 호스트 방화벽이 없음낮음수동

    계정 accounts.

    /etc/passwd와 /etc/shadow에서 직접 파싱합니다. /etc/shadow 읽기에는 root 권한이 필요하며, hostveil은 sudo로 자동 승격해 이를 얻습니다. 권한이 없으면 빈 비밀번호 점검은 건너뛰고 UID 0 점검은 계속 실행됩니다.

    ID표시 대상심각도수정
    accounts.uid0root가 아닌 계정이 root의 UID(0)를 가짐심각수동
    accounts.emptypassword로그인 계정에 빈 비밀번호가 설정됨심각수동

    파일 권한 fileperms.

    보안상 중요한 파일 목록의 권한을 확인해, 필요 이상으로 느슨하게 설정된 권한을 표시합니다.

    ID표시 대상심각도수정
    fileperms.shadow/etc/shadow가 0640보다 느슨함높음수동
    fileperms.passwd/etc/passwd가 root가 아닌 사용자에게 쓰기 가능높음수동
    fileperms.group/etc/group이 root가 아닌 사용자에게 쓰기 가능높음수동
    fileperms.hostkeySSH 호스트 개인 키를 root 외 사용자가 읽을 수 있음높음수동
    fileperms.sshd-configsshd_config가 root가 아닌 사용자에게 쓰기 가능보통수동

    이미지 CVE cve. (선택)

    Trivy가 설치되어 있으면, hostveil은 Compose 서비스가 실행하는 이미지를 스캔하여 알려진 심각·높음·보통 등급의 취약점을 찾습니다. 각 고유 취약점은 cve.<vulnerability-id> 형태의 ID를 가진 발견 항목이 되며 (예: cve.cve-2023-1234), 심각도는 Trivy의 등급에서 매핑됩니다.

    여기에 더해, 스캔된 이미지마다 cve.outdated-image 발견 항목이 하나씩 생겨 업스트림에 패치가 나온 취약점들을 요약합니다. 실제로 조치할 수 있는 건 이 항목입니다 — 수정 기능은 여기에 붙고, 개별 CVE 항목은 정보성으로 남습니다.

    ID탐지 내용심각도수정
    cve.outdated-image이미지에 업스트림 패치가 나온 취약점이 있음수정 가능한 것 중 최악검토
    cve.<vulnerability-id>한 이미지의 알려진 취약점 하나Trivy 기준수동
    개별 CVE에 수정 버튼이 없는 이유

    Trivy는 패키지가 3.0.11-1~deb12u2 버전에 도달해야 한다고 알려줍니다. 이건 이미지 안에 있는 OS 패키지 버전이지 이미지 태그가 아니고, 둘 사이를 잇는 매핑은 존재하지 않습니다 — 개별 CVE 수정을 만들려면 그 매핑을 지어내야 합니다. 이미지 단위 롤업은 이 문제를 아예 비껴갑니다. 이미 사용자가 고른 태그를 다시 받는 데는 버전 매핑이 필요 없고, 약속하는 것도 "태그를 다시 해석한다"까지일 뿐 특정 CVE가 사라진다고 말하지 않습니다. 다이제스트로 고정된 이미지에는 수정이 붙지 않습니다 — 다이제스트를 pull해봐야 아무것도 바뀌지 않기 때문입니다.

    "아직 패치가 없음"에 대한 솔직함

    아직 패치가 없다면, hostveil은 이를 고칠 수 있는 척하는 대신 사용 불가라는 정식 상태로 모델링합니다 — 스캔이 정직함을 유지하도록.
