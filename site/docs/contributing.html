    Contributing

    hostveil is Go, with a small dependency tree and a strict architecture. This is the short version; the canonical developer guide is docs/DEVELOPMENT.md.

    Prerequisites

    • Go — the version pinned in go.mod or newer.
    • git.
    • Optional: Vagrant + a provider (libvirt on Linux, VirtualBox on macOS/Windows) to run the demo VM.

    Build & test

    git clone https://github.com/seolcu/hostveil.git
cd hostveil
go build ./cmd/hostveil     # produces ./hostveil
go test ./...               # unit + fuzz + property tests

    Before sending a change, run the same checks CI runs:

    go build ./...
go vet ./...
gofmt -l .                  # must print nothing
go mod tidy                 # must leave go.mod/go.sum unchanged
go test -race ./...
    Linux tool, portable tests

    hostveil builds and its tests pass on any OS, but running the binary meaningfully needs Linux with the relevant tools present. Use the demo VM rather than running it against your own machine.

    Repository layout

    cmd/hostveil/        entry point + subcommand wiring
internal/
  core/              the shared engine — the only thing the UIs call
  check/             detection domains (compose, ssh, firewall, updates, cve)
  fix/ compose/ history/   fix registry, YAML editing, backup/rollback
  model/ platform/   pure value types; the OS/command seam
  ui/{cli,tui,web}/  thin UIs over the engine
demo/                the reproducible vulnerable-server VM (Vagrant)
site/                the marketing site + these docs (static)
docs/                developer documentation

    Architecture rule

    Enforced by a test

    UIs depend only on core. An import-lint test enforces that ui/* never imports fix, history, check, or compose. This is what keeps the TUI, web, and CLI behaving identically — they all go through the one engine.

    Adding a check

    Adding a detection domain means writing one package under internal/check/ that implements the Checker interface and registering it. Findings are created with a stable domain.rule ID, a severity, and a remediation kind (see What it checks and Fixing & rollback).

    Running it for real: the demo VM

    demo/ is a code-defined, deliberately vulnerable Ubuntu server. hostveil is built from your working tree inside the VM, so it always reflects your current code:

    cd demo
./run.sh up        # boot + provision + start the vulnerable stacks
./run.sh scan      # run hostveil against the server
./run.sh web       # dashboard at http://localhost:8787
./run.sh shell     # a shell on the server (then: sudo hostveil ...)
./run.sh halt      # shut it down

    The full walkthrough, per-platform provider setup, demo script, and reset workflow live in demo/README.md.
